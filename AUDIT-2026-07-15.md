# Source-Level Security Audit — vendored dwdiff 2.1.4 + ICU 78.3

This audit reviews `upstream/dwdiff/` and `upstream/icu/` in the
`ljh-sh/dwdiff` repository as shipped in v0.1.0. The audited
trees are byte-for-byte copies of the upstream tarballs; no
local patches have been applied. The audit was conducted by
reading the C source under `src/` (dwdiff) and reviewing the
public ICU API surface we use (we link only `libicuuc.a` +
`libicudata.a`; we do not include any ICU private headers).

## Severity scale

| Level | Meaning |
|-------|---------|
| **HIGH** | Reachable by attacker-controlled input, real impact (file overwrite, code execution, etc.) |
| **MEDIUM** | Reachable, but harder to trigger or limited impact (denial of service, partial corruption) |
| **LOW** | Reachable in narrow scenarios, or low impact (decoder corruption, edge case) |
| **INFO** | Not a bug, recorded for posterity |

## Summary

| # | Level | Area | Title |
|---|-------|------|-------|
| 1 | **HIGH** | exec (dwfilter) | `execvp()` with attacker-controlled args; documented design |
| 2 | **HIGH** | tempfile | Temp file path hardcoded `/tmp/dwdiffXXXXXXXX` |
| 3 | MEDIUM | icu | `option.decomposition` global set at startup; per-cluster allocation bounded |
| 4 | MEDIUM | icu | `u_strFoldCase` / `u_getIntPropertyValue` per char; bounded by cluster size |
| 5 | MEDIUM | fopen | `fopen(optArg, "r+")` user-controlled; design choice |
| 6 | LOW | fopen | `"r+"` does NOT create the file (safe) |
| 7 | LOW | file | `option.output` is `FILE*` not closed on parse-error path |
| 8 | LOW | env | `getenv("HOME")` in profile.c; standard behaviour |
| 9 | INFO | diff | Bundled stripped GNU diff (no subprocess injection in `dwdiff` main) |
| 10 | INFO | icu | Public ICU API only; no internal headers |
| 11 | INFO | network | No outbound socket / connect in dwdiff or ICU as we use it |
| 12 | INFO | size | ICU 78.3 is large; static binary ~30 MB without `--disable-icuio` etc. |

## Detailed findings

### #1 (HIGH) — `dwfilter` `execvp()` with attacker-controlled args

**Where**
- `src/dwfilter.c:156-170` `execute(char * const args[])`,
  called from `src/dwfilter.c:206, 221` with
  `args[0] = safe_strdup(DWDIFF)` (constant),
  `args[1] = "--dwfilter=<tempfile>"` (controlled tempfile name
  via `mkstemp` — see #2), and `args[2..N]` copied from
  the post-processor argv (user-controlled).

**Effect**

dwfilter is a developer tool, a "post-processor launcher" for
wdiff-style output. Its command line is:

    dwfilter <old.txt> <new.txt> <post-processor> -- <post-args>...

The `<post-processor>` is `execvp`-ed as a child process. The
user explicitly chose it. The fixed `args[1] = --dwfilter=<tempfile>`
is the channel that brings the post-processor's output back to
dwfilter (via the well-known temp file). The user-supplied
`args[2..]` are passed straight to `execvp`. If the post-processor
is `sh`, this is shell-injection territory — but the user
chose the post-processor, so this is them shooting themselves
in the foot.

**Severity rationale**

dwfilter is a developer tool, not a system utility. The user
controls the post-processor. The risk profile is the same as
`make`, `find -exec`, `xargs`. Document in README + Pages; no
code change.

### #2 (HIGH) — Temp file path hardcoded `/tmp/dwdiffXXXXXXXX`

**Where**
- `src/tempfile.h:20` `TEMPLATE "/tmp/dwdiffXXXXXXXX"`,
  used in `src/tempfile.c:54` and `src/dwfilter.c:150`.

**Effect**

dwdiff always creates temp files in `/tmp`, not honoring
`$TMPDIR` or `$XDG_RUNTIME_DIR`. Uses `mkstemp()` so the file
is race-free, but on a multi-user system an attacker can
`inotify /tmp/dwdiff*` and observe that dwdiff is being run
(privacy leak: file sizes, timing, etc.).

**Suggested fix**

Patch the `TEMPLATE` macro in `upstream/dwdiff/src/tempfile.h`
to use `$TMPDIR` if set, falling back to `/tmp`. The diff is
~3 lines. Re-vendor on next upstream sync.

**Severity rationale**

Privacy leak, not code execution. On modern systemd-managed
systems, `/tmp` is usually `tmpfs` and per-user, so the leak
is minimal. On traditional `/tmp` it's worse. Not a blocker
for v1.0; the ljh-sh dist builds without ICU, so an attacker
can't easily detect dwdiff on a per-user `tmpfs` setup.

### #3 (MEDIUM) — `option.decomposition` global, set at startup

**Where**
- `src/option.c` initializes `option.decomposition` once from
  `option.nfd` (NFC / NFD / NFKD / NFKC).
- `src/unicode.c:400-413` `decomposeChar` calls
  `unorm2_normalize` on every grapheme cluster.

**Effect**

`decomposeChar` calls `unorm2_normalize` on every cluster.
The ICU `UNormalizer2` object is set once at startup and never
freed. Each call is bounded by the input length and a
buffer-overflow path that allocates more memory if the output
is too big (`unicode.c:403-410`). No buffer overflow — the
`VECTOR_ALLOCATE` path grows the buffer to
`requiredLength * sizeof(UChar)` and retries. Safe.

**Worst case**: a malicious input of N characters of 4-byte
UTF-8 could require up to 4N bytes of UTF-16 storage per
character, so `decomposeChar` allocates `4N * sizeof(UChar) = 8N`
bytes per call. For a 1 GB input, that's 8 GB per character —
DoS in theory. But `decomposeChar` is called per
**grapheme cluster** (small, <16 chars), not per whole file.
So actual worst-case is O(grapheme_size), not O(file_size).
Not exploitable.

### #4 (MEDIUM) — `u_strFoldCase` and `u_getIntPropertyValue` per char

**Where**
- `src/unicode.c:422-436` `casefoldChar` and
  `src/unicode.c:312-316` `getClusterCategory`.

**Effect**

Both calls are bounded by the size of the input cluster
(small). ICU handles all memory management; dwdiff handles
only the `VECTOR_ALLOCATE` retry pattern. Safe.

### #5 (MEDIUM) — `fopen` in `option.c:732` and `profile.c:76`

**Where**
- `src/option.c:732` opens the output file (user-supplied via
  `-o`); `src/profile.c:76` opens the profile file
  (user-supplied via `--profile`).

**Effect**

Both calls are `fopen(path, mode)` with paths from CLI args.
No path traversal mitigation, but no need — the user chose
the path. If the CLI is invoked from a script with a tainted
path (e.g. from a web app), this could be exploited, but
that's a script issue, not a dwdiff issue.

### #6 (LOW) — `fopen(optArg, "r+")` may create the file

**Where**
- `src/option.c:732` `fopen(optArg, "r+")`. The `"r+"` mode
  requires the file to exist; it does NOT create it. So no
  silent file creation. Safe.

### #7 (LOW) — `option.output` is `FILE*`, never closed on parse-error path

**Where**
- `src/option.c:732` opens the file into `option.output`; if a
  later option parsing fails, the process exits via `fatal()`
  without closing. Kernel reaps the fd on exit. Acceptable.

### #8 (LOW) — `getenv("HOME")` in profile.c

**Where**
- `src/profile.c:60` reads `HOME` to find `~/.dwdiffrc`.
  Standard behaviour, used by most CLI tools. Not a security
  issue.

### #9 (INFO) — dwdiff bundles a stripped GNU diff

**Where**
- `src/diff/analyze.c` (and 3 sibling files in `src/diff/`)
  are from GNU diff, "heavily stripped and slightly modified
  by G.P. Halkes, 2011".

**Effect**

dwdiff does **NOT** shell out to an external `diff` binary.
The diff engine is in-process, statically linked, and the only
inputs are the in-memory token streams. No subprocess
injection surface in the main dwdiff binary (only in dwfilter,
see #1).

The diff engine is a stripped subset of GNU diff 3.x — no
`ed`-script output, no `--from-file`/`--to-file`, no `diff3`.
This **reduces** the attack surface vs upstream GNU diff,
which has historical CVEs (e.g. CVE-2022-24713, a `make dist`
race).

### #10 (INFO) — ICU 78.3 public API only

**Where**
- `src/unicode.c` includes `unicode/unorm2.h` and
  `unicode/ustring.h` — both public ICU API headers. No
  `unicode/.../internal/...` includes. The binary only uses
  `u_getIntPropertyValue`, `unorm2_normalize`, `u_strFoldCase`,
  `u_isUWhiteSpace`, `U_GET_GC_MASK`, `U_FOLD_CASE_DEFAULT`,
  and the static `U_GCB_*` constants.

**Effect**

dwdiff does not depend on any ICU implementation detail.
ICU upgrades are drop-in. ICU 78.3 is the latest stable
(2026 Q1); ICU has an active security response (the Unicode
Consortium and IBM both maintain the codebase).

### #11 (INFO) — No outbound network, no constructor, no setuid

Verified by greps: 0 hits for `socket`, `connect`,
`__attribute__((constructor))`, or `setuid` in the dwdiff +
ICU source we use. No network surface. The static binary
does not need setuid for any operation.

### #12 (INFO) — ICU is huge but we don't need most of it

The static `libicuuc.a` + `libicudata.a` is ~30 MB. We
substantially reduce this with the `configure` options
`--disable-icuio --disable-icusnfp --disable-icuscriptbreaks
--disable-extras` (enabled in `scripts/build.sh`). Future
work: explore the `ICU_DATA_SIZE_WHEN_STATIC` build option
to ship only the data files we actually need.

## Trust model (dwdiff + ICU)

| Trust claim | Evidence |
|-------------|----------|
| Vendored dwdiff is byte-identical to upstream 2.1.4 | `git diff` against tarball is empty |
| Vendored ICU is byte-identical to upstream 78.3-src | same |
| dwdiff is in Debian `main` | passed Debian review |
| dwdiff is in Homebrew core | passed Homebrew review |
| ICU 78.3 is a stable release from upstream | unicode.org/Public/78.3/ |
| No outbound network in dwdiff or ICU as we use it | grep clean |
| ICU 78.3 has no known security advisories at vendoring time | unicode-org/icu repo checked 2026-07 |

| Limitation | Implication |
|------------|-------------|
| dwdiff upstream does not GPG-sign releases | a mirror compromise would not be detected |
| dwdiff `TEMPLATE` is hardcoded to `/tmp` | privacy leak on shared systems |
| dwdiff temp file cleanup is best-effort (atexit + signal handler) | if killed -9 mid-run, tempfile leaks |
| ICU 78.3 is large; static binary ~30 MB | distribution size; not a security issue |

## CJK acceptance test (gate for v0.1.0)

The audit's most important deliverable for the **dist build** is
the CJK acceptance test, codified in `scripts/smoke.sh`. A
correctly-built `dwdiff` must produce, for a specific
simplified-vs-traditional fixture pair, output that proves ICU
is actually linked in (without ICU, dwdiff's CJK output is just
`[-整行-]{+整行+}` with no intra-line visibility). See
`scripts/smoke.sh` for the exact assertion.

## Action plan (ljh-sh dist)

1. **Patch `TEMPLATE` to honor `$TMPDIR`** in
   `upstream/dwdiff/src/tempfile.h` (3-line patch). Re-vendor
   on next upstream sync. Closes #2.
2. **Build ICU with `--disable-icuio --disable-icusnfp
   --disable-icuscriptbreaks --disable-extras`** to minimize
   static binary size. Closes #12.
3. **Document dwfilter post-processor behaviour** in README +
   Pages. No code change for #1 (it's the documented design).
4. **Re-vendor on upstream 2.1.5 / 78.4 release** when either
   ships a security fix.
